1. URIs should never be changed. ❌
  · 理论上应该尽量保持稳定，但有时不得不改变（如业务变化），用重定向解决。
2. URIs must be constructed by the client. ❌
  · 客户端不需要构造URI，服务器应提供 URI（如通过超媒体）。
3. URIs should be short in length. ✅
  · 简短 URI 更容易记忆和输入，但不是强制，是一种推荐实践。
4. URIs should be case-sensitive. ✅
  · 实际上 URI 是大小写敏感的（除协议、主机名部分外）。
5. HTTP verbs should be used instead of operation names in URIs. ✅
  · REST 最佳实践：用 HTTP 方法（GET, POST, PUT, DELETE）表示操作，不在 URI 中加入动作动词（如 /deleteItem）。
6. Use spaces when designing a URL. ❌
  · 不允许空格，应该用 %20 或连字符。
7. Redirection must be used if a change in URI is required. ✅
  · 当 URI 必须改变时，应该用重定向（如 301 Moved Permanently）。

1. 题意
有一个 Sender 接口，定义了一个 Send(Message) 方法。
有一个 Process(sender Sender) 函数，它接受一个实现了 Sender 接口的参数。
有一个独立的 Send(Message) 函数，它不是方法。
问题是：如何调用 Process 函数，传入 Send 函数（而不是接口实现）？
---
2. Go 语言中的接口与适配
Process 需要一个类型，这个类型有 Send(Message) 方法。
而 Send 是一个普通函数，不是方法，所以不能直接当作 Sender 传给 Process。
要让函数适配接口，常用的方法是定义一个函数类型，并让该函数类型实现接口的方法，方法内部调用函数本身。
---
3. 选项分析
---
选项 1：Use Process(Send)
❌ 不行。
Send 是一个函数，不是 Sender 类型，不能直接传给需要 Sender 接口的地方。
---
选项 2：Define a new struct type, MySender, with a method func (m MySender) Send(Message) that calls Send function.
✅ 可行。
这是最直接的方法：定义一个空结构体 MySender，给它加一个 Send(Message) 方法，方法内部调用全局函数 Send。
然后可以创建 MySender{} 传给 Process。
---
选项 3：Define a function type SendFunc as func(Message), then define a method (s SendFunc) Send(m Message) return s(m), and call Process(SendFunc(Send)).
✅ 可行。
定义 type SendFunc func(Message)，它作为函数类型。
让 SendFunc 类型实现 Sender 接口的方法 Send(m Message) { s(m) }。
然后 SendFunc(Send) 把函数 Send 转换成 SendFunc 类型，这个转换后的值有 Send 方法，符合接口，可以传给 Process。
这是 Go 里很常见的适配函数到接口的方式。
---
选项 4：Define a function type SendFunc as func(Message), then define a method (s SendFunc) Send(m Message) return s.Send(m) and call Process(SendFunc(Send)).
❌ 不可行。
s 是函数类型 func(Message)，没有 Send 方法，所以 s.Send(m) 是错误的。
应该是 s(m) 才对。
这里写法错误。

正确可行的：选项 2 和 选项 3。

Go 中：
· var err *customError 是 nil 指针。

1. 代码分析
type Node 有 Children []*Node（存的是指针）。
函数 NewChild(node Node) Node：
· 创建一个新 child 节点（局部变量）
· node.Children = append(node.Children, &child)：将 child 的地址加入 node.Children。
· 返回 child（按值返回）。
main 中：
· root := Node{}：root 是一个空节点，Children 是 nil。
· child := NewChild(root)：把 root 传进函数。
· 打印 len(root.Children)。
---
2. 传值问题
Go 中函数参数是传值。
NewChild(root) 传的是 root 的副本（值拷贝）。
在函数内部修改 node.Children 只是在副本上修改，不会影响 main 里的 root.Children。
所以 len(root.Children) 打印 0（原 root 没变）。
---
3. 函数语义
注释说：“Create a new child node and return that child”
· 它确实创建了新节点并返回了它（按值返回）。
· 但返回的是副本，不是原始 child 变量，这不影响使用（可以正常用）。
· 更大的问题是：它没把 child 真正加到 root 里。
---
4. 正确修改
为了让 root.Children 包含 child 指针，需要传入 *Node：
```go
func NewChild(node *Node) Node {
   child := Node{}
   node.Children = append(node.Children, &child)
   return child
}
```
这样传入 root 的地址，修改会生效。
或者返回 *Node 也可以，但那样是返回 child 的地址，不是 child 副本。
---
5. 选项对应
打印 0 因为传值。
修改方法：NewChild(*Node) Node 或 NewChild(*Node) *Node。
选项里：
· 打印 0 + 改成 NewChild(*Node) Node → 选项 3
· 打印 0 + 改成 NewChild(*Node) *Node → 选项 4
看哪个符合题意：
“It prints 0 because root is passed by value. The function should be declared as NewChild(Node) Node.” ✅
“It prints 0 because root is passed by value. The function should be deleted as NewChild(Node) *Node.” ❌（deleted 是 typo，应是 declared，且返回 *Node 也行，但注释要求返回 child，不是必须返回指针）
更精确的是 选项 3（返回 Node 即可，因为返回的是 child 副本，不影响使用，只是 root 指针需要修改）。

---
1. 函数逻辑
```go
func splitAndProcess(values ...int) int {
   lowerHalf := values[0 : len(values)/2]
   upperHalf := values[len(values)/2:]
   y := process(append(lowerHalf, 10))
   x := process(append(upperHalf, 10))
   return x + y
}
```
· lowerHalf 和 upperHalf 是 values 切片的两个子切片，共享同一底层数组。
· append(lowerHalf, 10)：
 · 如果 lowerHalf 长度 < 容量，直接在底层数组 lowerHalf[len(lowerHalf)] 位置写入 10，并返回新切片。
 · 这可能会覆盖 upperHalf[0] 原本的值，因为它们共享底层数组。
---
2. 举例
假设 values := []int{1, 2, 3, 4}，len=4。
· lowerHalf = values[0:2] → [1,2]，cap=4（因为来自 values）
· upperHalf = values[2:4] → [3,4]，cap=2
执行 append(lowerHalf, 10)：
lowerHalf 长度 2，容量 4，所以原地 append，在下标 2 写入 10，底层数组变成 [1,2,10,4]，返回 [1,2,10]。
但 upperHalf 还是 [3,4] 吗？
不，底层数组下标 2 已改成 10，所以 upperHalf 现在指向的底层数组是 [10,4]，即 upperHalf 变成了 [10,4]。
然后 append(upperHalf, 10)：
upperHalf 现在是 [10,4]，长度 2，容量 2（因为子切片从 index=2 开始，容量是原数组长度 - index = 4-2=2），所以 append 需要扩容，分配新数组，不会影响 lowerHalf。
但第一个 append 已经破坏了 upperHalf 原本的值。
---
3. 问题所在
· 第一个 append 修改了 lowerHalf 的底层数组，这个修改影响了 upperHalf 的第一个元素。
· 第二个 append 由于 upperHalf 容量不够，会分配新数组，不影响原数组，但第一个 append 已经造成数据错误。
所以 upperHalf 在被传给 process 时已经和预期不同。
